## 二、Momenta 实习（DDOD / OCC）——深度拆解

### 2.1 一句话项目介绍

> DDOD 是自动驾驶场景下的**数据生产流水线平台**，支撑内部数据任务的创建、编排、调度与监控。我负责 **OCC 工作流与任务管理模块**后端，用 SpringBoot + MyBatis，完成 Union/Occ/Task 的 REST API、批量任务提交链路、监控子系统和统一数据访问层。

### 2.3 职责点 → 技术细节 → 面试话术

#### ① Union / Occ / Task 核心 REST API

**你做了什么：** 任务创建、查询、状态流转、外部系统对接。

**必会细节：**

- REST 资源设计：`/unions`、`/occs`、`/tasks`；用名词复数；状态变更用 `PATCH` 或专用 action。
- 幂等：创建任务接口如何防重（业务唯一键 / Idempotency-Key / MD5 指纹）。
- 状态机：草稿→已提交→运行中→成功/失败/取消；非法跳转要拒绝。
- 外部系统集成：超时、重试、熔断、签名鉴权、回调验签。

**高频追问：**

1. 为什么用 REST 而不是 RPC？  
   → 内部平台、多语言调用方、资源型 CRUD + 状态流转，REST 更合适；高性能内部链路可再上 RPC。
2. 状态流转如何保证一致性？  
   → DB 事务 + 乐观锁（version）或状态条件更新 `WHERE status=?`；必要时发领域事件。

#### ② UnionOccTaskService：解析 → 批量 MD5 → 并发建 workflow → 落库 → 刷新状态

这是**最容易被深挖**的一条链路。

**建议口述流程：**

1. 接收数据集清单 / 文件列表  
2. 解析元数据（路径、数量、类型）  
3. **批量计算 MD5**：用于去重、完整性校验、可追溯指纹  
4. **并发创建 workflow**：线程池提交，控制并发度，避免打爆下游  
5. 任务信息落库（关联 Union/Occ/Task）  
6. 状态刷新：轮询或回调驱动 monitor

**必会细节：**

| 点 | 要能说到 |
|----|----------|
| 批量 MD5 | 大文件如何流式算 MD5；线程池并行；失败重试；结果写回 |
| 并发创建 | `ExecutorService` / 线程池；队列长度；拒绝策略；下游限流 |
| 可追溯 | 任务 ID、父工作流 ID、MD5、提交人、时间、状态变更日志 |
| 效率提升 | 「显著提升批量提交效率」→ 准备一个量级说法：如从串行 N 次变成并发 K 路，耗时近似 /K（受 IO 限制） |

**高频追问：**

1. 并发创建 workflow 如何保证不重复、不丢？  
   → 唯一键（数据集指纹+版本）；事务提交；失败重试 + 幂等。
2. MD5 冲突怎么办？  
   → 实际碰撞极低；业务上可 MD5 + 文件大小 + 路径 组合唯一。
3. 线程池参数怎么设？  
   → IO 密集：核心线程可偏大；绑定队列；拒绝策略用 CallerRuns 或丢弃并告警；监控活跃线程与队列积压。

#### ③ 独立完成 monitor 监控子系统

**能力点：** 任务进度同步、部署状态刷新、成本计算、告警前置判断。

**必会细节：**

- 进度同步：定时拉取 vs 事件推送；最终一致；避免全表扫。
- 成本计算：算力时长 × 单价、存储量、调用次数；注意时区与计费口径。
- 告警前置：阈值（失败率、卡住时长、成本突增）；降噪（静默期、聚合）。
- 可观测：日志、指标、链路追踪三件套（即使实习只做了业务监控，也要知道）。

**高频追问：** 如何发现「卡住的任务」？  
→ 状态=运行中 且 `updated_at` 超时；或心跳超时；进入告警与人工介入。

#### ④ ORMMethodService 数据访问门面 + upsert

**设计意图：** 多张 DPL 表的 upsert/查询逻辑统一，避免各处复制 SQL。

**必会细节：**

- 门面模式（Facade）：对上提供业务语义方法，对下封装 MyBatis Mapper。
- upsert：MySQL `INSERT ... ON DUPLICATE KEY UPDATE` 或先查后写（注意并发）。
- 好处：一致性、可测试、后续换表结构只改一处。

**高频追问：** upsert 并发安全吗？  
→ 依赖唯一索引；高并发下可能用分布式锁或 DB 唯一约束吃冲突后重试。

#### ⑤ 多数据源联调

**必会细节：**

- 为何多数据源：业务库 / 流水线元数据库 / 外部同步库分离。
- Spring 动态数据源：`@DS` 或 `AbstractRoutingDataSource`。
- **事务不能跨库轻易 `@Transactional`**：要么单库事务，要么最终一致 + 补偿。
- 联调问题：连错库、时区、字符集、只读从库延迟。

## 三、智农云眸（智慧农业全栈平台）——最高频后端考点

### 3.1 一句话项目介绍

> 智农云眸是面向智慧农业 C/B 端的全栈平台，产品侧有多模态病虫害识别、RAG 专家问答、Multi-Agent 调度。我重点做 **高并发链路、缓存一致性、短链接生成与分片**，技术栈 SpringBoot + Redis + MySQL + ShardingSphere + RocketMQ + Sentinel。

### 3.2 缓存一致性：「先更新 DB，再删缓存」+ 分布式重试

#### 场景

大模型对话**上下文（Context）**会被频繁读写：多端交互、消息追加、会话切换。

#### 方案（Cache-Aside 删缓存版）

```
写请求：
  1. 更新数据库（权威数据）
  2. 删除 Redis 中对应 Context Key
  3. 若删缓存失败 → 重试 / 投递延迟删除消息（分布式重试）

读请求：
  1. 读缓存，命中则返回
  2. 未命中则读 DB，回填缓存
```

#### 为什么「删」而不是「更新缓存」？

| 策略 | 问题 |
|------|------|
| 先更新缓存再更新 DB | DB 失败导致缓存脏 |
| 先更新 DB 再更新缓存 | 并发下两个写请求可能把旧值写回缓存 |
| **先更新 DB 再删缓存** | 读时回源，最终一致；短暂可能打到 DB |

#### 延迟双删（加分项，简历没写但建议会）

1. 更新 DB  
2. 删缓存  
3. 休眠几十到几百 ms（覆盖主从延迟/并发读回填窗口）  
4. 再删一次缓存  

#### 分布式重试怎么做？

- 本地重试失败 → RocketMQ / 延迟队列投递「删除 Key」消息  
- 消费者幂等删除  
- 监控删除失败率  

#### 最终一致性话术

> 我们接受短时间窗口内缓存可能旧，但通过删缓存 + 重试保证最终与 DB 一致；大模型上下文场景更看重正确会话内容，而不是强一致毫秒级。

**高频追问（含答案）：**

**Q1：缓存穿透 / 击穿 / 雪崩分别是什么？怎么治？**  
**A：**  
- **穿透**：查根本不存在的数据，缓存没有、DB 也没有，每次都打到 DB。治理：布隆过滤器拦截；或缓存空值并设短 TTL。  
- **击穿**：某个热点 Key 过期瞬间，大量并发同时打 DB。治理：互斥锁重建（`synchronized` / Redis 锁）；或逻辑过期（Key 不过期，值里带过期时间，异步刷新）。  
- **雪崩**：大量 Key 在同一时间过期，或 Redis 宕机，流量整体打到 DB。治理：TTL 加随机抖动；多级缓存；Redis 主从/集群高可用；入口限流（Sentinel）。

**Q2：大 Key 有什么危害？怎么拆？**  
**A：** 大 Key 会导致：网络传输慢、阻塞其他命令、主从同步延迟、内存倾斜、删除时卡顿。治理：拆成多个小 Key；Hash 大字段拆分；短链场景用 `短码 → URL` 的 String，不要把无关数据塞进同一个巨型结构；删除用渐进删除（UNLINK）。

**Q3：Redis 持久化 RDB vs AOF？**  
**A：** RDB 是定时快照，恢复快、文件小，但可能丢最后一次快照后的数据。AOF 记录写命令，丢数据更少（可 everysec），文件更大、恢复更慢。生产常「RDB + AOF」混用；对短链发号、会话上下文等，更关心少丢数据，倾向开启 AOF。

**Q4：分布式锁用 Redis 怎么实现？红锁有什么争议？**  
**A：** 常用 `SET key token NX EX seconds` 加锁，业务结束用 Lua 校验 token 再删，避免误删别人的锁。注意：锁续期（看门狗）、超时要大于业务耗时。RedLock（多实例红锁）争议在于：时钟漂移、网络分区下安全性存疑；多数业务用「单 Redis + 主从」或直接用 Redisson 看门狗即可，强一致场景更推荐 ZooKeeper/etcd。

### 3.3 高并发网址跳转：多层缓存 + synchronized，防击穿/穿透

#### 业务

短链访问：`短码 → 原始长链`，读多写少、热点明显。

#### 设计要点（对齐简历）

1. **多层判断**：本地缓存（可选）→ Redis → DB  
2. **synchronized / 互斥**：同一短码只有一个线程回源，其它等待，防击穿  
3. **避免大 Key**：不要把无关数据塞一个巨型 Hash；短码→URL 用 String 或合理分片 Hash  
4. **空值缓存**：防穿透  

**口述模板：**

> 跳转链路对延迟敏感。我们先查 Redis；未命中时用短码粒度的锁（进程内 synchronized 或 Redis 锁）保证只有一个请求查库回填，避免热点短链过期时打爆 MySQL；同时对非法短码缓存空值，防止穿透。

**注意：** 多实例部署时，仅 JVM `synchronized` **不能**跨节点互斥，要补 Redis 分布式锁或 Redisson。面试被问到要主动说清边界。

### 3.4 短链接基因法分片（ShardingSphere）——必须画图能讲

#### 背景问题

短链表很大，要分表；但又要同时支持：

- 按 **短链接 code** 查询（跳转主路径）  
- 按 **gid**（分组/用户维度）查询（管理端）

若只按 code 哈希分表，按 gid 查询会变成**扫所有分表**。

#### 基因法核心思想

把 **分表路由信息（基因）编码进短链本身**，使得：

- 拿到短链 → 直接解析出分片位 → 定位唯一分表  
- gid 与短链共享同一套分片基因 → gid 查询也能路由到正确分表  

示意：

```
gid ──► 取低 N bit 作为「分片基因」
             │
短链生成时 ──┼──► 把基因嵌入短码（或固定位）
             │
查询短链 ──► 从短码提取基因 ──► ShardingSphere 路由到 ds_x.table_y
查询 gid  ──► 同样用基因路由，避免全分片扫描
```

**面试要能说的点：**

1. 分片键是什么？基因多少 bit？分表数量 = 2^N 的关系。  
2. 短码字符集（Base62）与基因位如何映射。  
3. 扩容怎么办？（预留基因位 / 双写迁移——了解即可）  
4. 「性能提升近两倍」：准备前后对比口径（QPS、P99、或扫描分表数从 N→1）。

**高频追问：** ShardingSphere 分库分表后全局唯一 ID 怎么做？  
→ 雪花算法、号段模式（正好接到下一节 Redis 发号）。

### 3.5 短链生成：Redis 批量发号 + AtomicLong + 号段越界校验

#### 目标

分布式多机部署下：**高速、无重复、可水平扩展**。

#### 号段模式（Leaf 思想）

```
每个服务实例：
  1. 向 Redis（或 DB）申请一段号：[10001, 20000]
  2. 本地用 AtomicLong 自增发号（无网络往返）
  3. 接近段尾 → 异步再申请下一段
  4. 越界校验：超过 max 则拒绝/重新取号段
```

**为什么快：** 发号主路径在内存；Redis 只在「取号段」时访问，吞吐随实例数近似线性升。

**为什么唯一：** 号段全局不重叠（Redis INCRBY / Lua 保证分配原子性）。

**与短链编码：** 数字 ID → Base62 → 再嵌入基因位 → 最终短码。

**高频追问：**

1. Redis 挂了怎么办？ → 本地剩余号段仍可发；新号段申请失败则降级/告警；持久化用 AOF。  
2. AtomicLong 会超发吗？ → 每次自增前检查 `current <= segmentEnd`。  
3. 和雪花算法对比？ → 号段趋势递增、本地快；雪花含时间与机器位、不依赖号段中心，但受时钟回拨影响。

## 四、温室巡检机器人（银黄农场）——算法/机器人深度

### 4.1 一句话项目介绍

### 4.2 巡检方案对比（面试官爱听「为什么选这个」）

| 方案 | 原理简述 | 温棚无轨环境优缺点 | 适用边界 |
|------|----------|--------------------|----------|
| OpenCV 巡线 | 颜色/边缘提取固定引导线 | 实现简单；依赖铺线/光照，无轨难用 | 有轨或强视觉标记 |

| 算法 | 类型 | 特点 | 为何对比 |
|------|------|------|----------|

**你简历关键动作：** TEB 上增加**作物安全距离约束**（代价函数/膨胀层），防止蹭苗。

## 八、按「面试追问链」自测清单（问题 + 参考答案）

> 用法：先遮住答案口述，再对照。答案按「可直接说给面试官听」的口吻写。

### 8.1 后端主链

**Q1：缓存穿透、击穿、雪崩分别是什么？你在短链跳转里怎么做的？**  
**A：** 穿透是查不存在的数据一直打到 DB；击穿是热点 Key 过期瞬间并发打 DB；雪崩是大量 Key 同时失效或缓存挂了。跳转链路里我做了多层判断：先 Redis，未命中再回源；对非法短码缓存空值防穿透；对热点短码用互斥锁（`synchronized`/分布式锁）只让一个线程重建缓存防击穿；TTL 加随机、入口用 Sentinel 限流降低雪崩风险。

**Q2：为什么采用「先更新数据库再删除缓存」？会不会出现脏数据？延迟双删、分布式重试怎么做？**  
**A：** 双写缓存容易在并发下把旧值写回去；删缓存更安全——下次读 miss 再从 DB 回填。短暂窗口内可能读到旧缓存或打到 DB，我们接受最终一致。为缩小脏窗：更新 DB 后删缓存；失败则本地重试，再失败丢到 MQ/延迟队列异步删（幂等）；必要时做延迟双删（删一次 → 等几十到几百毫秒 → 再删），覆盖主从延迟和并发回填窗口。大模型上下文场景更看重会话正确性，不强求毫秒级强一致。

**Q3：什么是基因法分片？为什么短链要用？gid 和短码怎么都能路由到正确分表？**  
**A：** 普通按短码哈希分表后，按 gid 查询会扫全部分表。基因法把分片信息（如 gid 低 N bit）编码进短码固定位：跳转时从短码解析基因直接路由；管理端按 gid 查询时用同一套基因路由，避免全分片扫描。分表数通常与基因位数相关（约 2^N）。所以查询性能能明显提升（简历口径近两倍）。

**Q4：短链 ID 如何保证分布式下高速且不重复？Redis 批量发号 + AtomicLong + 号段越界是怎么配合的？**  
**A：** 用号段模式：实例向 Redis 原子申请一段号（如 1 万个），本地用 `AtomicLong` 自增发号，主路径无网络；快用完再异步取下一段；每次发号前校验不超过段尾，越界则重新取段。号段全局不重叠所以唯一；多机各自持有不同号段，吞吐近似随机器数线性扩展。数字 ID 再 Base62，并嵌入基因位生成最终短码。

**Q5：多实例部署时只用 synchronized 够不够？不够怎么升级？**  
**A：** 不够。`synchronized` 只在单 JVM 内互斥，多实例仍可能同时回源打 DB。应升级为 Redis 分布式锁：`SET key token NX EX`，业务结束用 Lua 校验 token 删除；或用 Redisson 看门狗自动续期。锁粒度要细到单个 shortCode，避免全局大锁。

**Q6：RocketMQ 在你的项目里可以落在哪些点？至少讲一个完整链路。**  
**A：** 典型落点是「删缓存失败的异步重试」：本地删 Redis 失败 → 发延迟/普通消息（Key、版本）→ 消费者幂等删除并监控失败率。也可用于短链创建后的异步统计、告警通知。注意消费幂等（唯一键/状态判断），避免重复删或重复计费。

**Q7：Sentinel 的限流、熔断怎么和短链入口结合？**  
**A：** 短链跳转是公开高频入口，对跳转接口按 QPS/线程数限流，保护 Redis 和 DB；当错误率或慢调用比例过高时熔断，快速失败并返回友好页，避免雪崩。可和缓存击穿治理叠加：限流挡洪峰，锁重建挡热点。

**Q8：Momenta 里批量 MD5 + 并发创建 workflow，如何保证不重复、不丢失？线程池怎么考虑？**  
**A：** 不重复：业务唯一键（数据集指纹/MD5+版本等）+ 幂等创建；DB 唯一约束挡并发双插。不丢失：事务落库；失败重试；状态可查询可补偿。线程池按 IO 密集配置，有界队列，拒绝策略可用 CallerRuns 或失败告警，并限制并发度避免打爆下游；活跃线程和队列积压要可观测。

**Q9：upsert 是什么？并发安全吗？ORMMethodService 门面模式有什么好处？**  
**A：** upsert 是存在则更新、不存在则插入（如 `ON DUPLICATE KEY UPDATE`）。并发安全依赖唯一索引；冲突时以 DB 约束为准并重试。门面把多张 DPL 表的 upsert/查询收口到统一服务，上层只调业务方法，避免 SQL 复制，改表结构只改一处，一致性和可维护性更好。

**Q10：多数据源场景下 `@Transactional` 要注意什么？**  
**A：** Spring 事务默认绑在单个 DataSource 上，不能想当然跨库强一致。做法：单库内本地事务；跨库用最终一致（消息 + 补偿）或分布式事务（成本高，一般不首选）。联调还要防连错库、主从延迟读旧数据、时区字符集不一致。

### 8.2 算法 / 机器人链

**Q11：OpenCV 巡线、YOLO 路面分割、视觉 SLAM、激光 SLAM 怎么对比？无轨温棚怎么选？**  
**A：** OpenCV 巡线实现简单，但依赖铺线/稳定光照，无轨难用。YOLO 分割可行驶区域，适应性较好，需数据和算力。视觉 SLAM 成本低，但温棚重复纹理、光照变化易丢跟踪。激光 SLAM 通常更稳，成本和遮挡仍是问题。我们做了方案对比与适用边界分析，并重点优化视觉 SLAM 在农业场景的可用性。

**Q12：「定位精度从约 25% 提升到约 70%」「路径跟踪误差 &lt;15cm」分别怎么理解？怎么测？**  
**A：** 前者建议口径说成「定位/跟踪成功可用比例」或「满足误差阈值的帧/路程占比」（面试前按你真实定义统一）。后者是行间自主巡检时，实际轨迹相对规划路径的偏差，银黄农场约 10m 行间实测小于 15cm，可用卷尺/轨迹对比/RMSE 统计。回答时主动说明指标定义，避免被追问含糊。

**Q13：A*、RRT*、Hybrid A*、TEB 一句话区别？你为什么用 Hybrid A* + TEB？**  
**A：** A* 是网格搜索，短路径最优但未必满足车辆运动学。RRT* 是采样规划，适合复杂空间，渐近最优。Hybrid A* 在搜索中考虑朝向和转弯半径，轨迹更可执行，适合移动机器人。TEB 是局部规划，优化时空轨迹做跟随时避障与速度平滑。我们全局用 Hybrid A*，局部用 TEB，并在代价/膨胀里加作物安全距离，减少蹭苗。

**Q14：作物安全距离在 ROS 里具体落在哪？**  
**A：** 落在 costmap 的膨胀层（inflation）：把障碍/作物区域向外膨胀安全半径，规划器避开高代价区域；也可在 TEB 约束里加大障碍权重。本质是「别贴苗走」。

**Q15：ROS Navigation 大致由哪些模块组成？**  
**A：** 定位（AMCL/SLAM 位姿）+ 地图/代价地图 costmap + 全局规划器 + 局部规划器 + 控制器输出速度到电机；`move_base`（或 Navigation2）把这些串起来。我们把 Hybrid A* / TEB 集成进该框架做实地联调。

### 8.3 小程序 / 端侧链

### 8.4 大模型链

**Q20：RAG 的基本五步是什么？农业场景怎么用？**  
**A：** 文档切分 → Embedding → 向量检索 TopK →（可选）Rerank → 拼进 Prompt 生成。农业上可检索种植手册、病虫害知识、用药说明，降低幻觉，支撑「专家问答」。

**Q21：为什么 Prompt / 上下文要用 Redis 做缓存和动态管理？**  
**A：** 系统提示、模板、会话 Context 读写频繁；Redis 低延迟，可缓存 Prompt 降本降时延，并按会话 Key 管理上下文；再配合「改 DB 删缓存」保证多端交互后最终一致。

**Q22：Tool Calling / Multi-Agent 和普通多轮聊天有什么区别？**  
**A：** 普通聊天只生成文本。Tool Calling 是模型选择工具 → 执行（查库/调 API）→ 观察结果 → 再回答，能闭环办事。Multi-Agent 是多个角色分工（如识别、检索、规划）协同调度，适合病虫害识别 + 问答 + 任务编排这类复合流程。

**Q23：LLM/VLM 微调你怎么答才稳（简历写的是了解）？**  
**A：** 坦诚是了解思路：SFT 监督微调；LoRA/QLoRA 低成本适配；VLM 做图像+文本（病虫害图问答）。强调数据质量比盲目堆参数更重要；我的强项是把模型接到业务（RAG、端侧推理、农业场景），不是大规模预训练。

### 8.5 综合 / 场景题（加分）

**Q24：如果短链服务 Redis 突然宕机，系统会怎样？你怎么兜底？**  
**A：** 跳转会大量回源 DB，有打垮风险。兜底：本地/多级缓存短暂顶住；Sentinel 限流；号段模式本地剩余号段仍可发号；Redis 恢复后重建缓存；告警与降级页。持久化开 AOF 降低丢号段元数据风险。

**Q25：如何向面试官 1 分钟讲清「智农云眸你做的四件事」？**  
**A：** 一是大模型上下文用「改 DB 删缓存 + 重试」保最终一致；二是短链跳转多层缓存 + 互斥重建，治穿透击穿；三是 ShardingSphere 基因法分片，短码和 gid 都能精准路由；四是 Redis 号段 + 本地 AtomicLong 高速无重复发号，支持水平扩展。

**Q26：Momenta 和智农云眸的技术共通点是什么？**  
**A：** 都是后端工程：高并发下的正确性（幂等、状态、缓存一致）、可观测（监控告警）、可维护（门面/分层）。一个偏数据流水线任务编排，一个偏 C 端高并发读与分片发号。
